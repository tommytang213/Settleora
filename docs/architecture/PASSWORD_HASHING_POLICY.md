# Password Hashing Policy

This document defines Settleora's local-account password hashing policy. It is an architecture policy for the internal hashing boundary and future credential workflows. Design-only credential creation, password verification, and rehash workflow boundaries are defined in [AUTH_CREDENTIAL_WORKFLOW_DESIGN.md](AUTH_CREDENTIAL_WORKFLOW_DESIGN.md). It does not authorize migrations, OpenAPI changes, generated clients, login behavior, credential creation behavior, or UI behavior.

The policy is based on current OWASP Password Storage Cheat Sheet guidance and NIST SP 800-63B-4 password authenticator guidance. Future auth workflow work must re-check those sources and benchmark deployment hardware before freezing production parameters.

## References

- [OWASP Password Storage Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html)
- [NIST SP 800-63B-4](https://pages.nist.gov/800-63-4/sp800-63b.html)

## Current State

- The schema includes `local_password_credentials`.
- `local_password_credentials` is linked to `auth_accounts` and has fields for password hash material, algorithm name, algorithm version, algorithm parameters, status, timestamps, last verification time, revocation time, and `requires_rehash`.
- An internal password hashing service boundary exists for Argon2id verifier creation, verification, metadata, and rehash decisions.
- Login, credential creation, password rotation, reset, recovery, current-user, and auth session runtime endpoints do not exist.
- No OpenAPI auth paths, generated auth clients, or UI auth behavior exist.

## Algorithm Policy

Settleora should prefer Argon2id for new local-account password verifiers.

Argon2id is the preferred policy because it is designed for password hashing and provides memory-hard resistance to offline attacks. The internal implementation uses Geralt for encoded Argon2id verifiers. Future auth workflow work must tune memory, iteration, and parallelism parameters on deployment-class hardware.

Fallback policy is intentionally narrow:

- If Argon2id is unavailable in an acceptable .NET dependency, evaluate another modern password hashing option only through a separate implementation design.
- If a deployment requires FIPS-compatible password hashing, use PBKDF2-HMAC-SHA-256 with an approved implementation and a policy-configured iteration count.
- PBKDF2 is a compatibility fallback, not the preferred default for non-FIPS deployments.
- The current internal service uses Geralt for Argon2id. Future library changes must go through implementation design review.

## Stored Hash Format

The `password_hash` field must store an encoded password verifier output. It must never store a plaintext password, reversible encrypted password, or application-specific secret.

Stored verifier data must be sufficient to verify an existing password and migrate it later:

- `password_hash` stores the encoded verifier output produced by the approved hashing implementation.
- `password_hash_algorithm` stores the algorithm family, such as `argon2id` or `pbkdf2-hmac-sha256`.
- `password_hash_algorithm_version` stores Settleora's policy version for that algorithm and parameter profile.
- `password_hash_parameters` stores normalized, non-secret parameters required to verify and migrate the verifier, such as memory cost, iteration count, parallelism, derived-key length, PRF, salt encoding location, and optional pepper key identifier.
- `status` controls whether a credential is active, disabled, or revoked.
- `last_verified_at_utc`, `revoked_at_utc`, `updated_at_utc`, and `requires_rehash` support verification history, revocation, and future migration flows.

Prefer a standard self-describing verifier encoding when the selected library supports one, such as PHC-style encoding for Argon2id. If the verifier encoding already includes salt and work factors, `password_hash_parameters` should still record normalized policy metadata so Settleora can query, audit, and migrate credentials without parsing every verifier string ad hoc.

## Salt Policy

Every password verifier must use a unique random salt for that verifier. Reusing a salt across credentials or password rotations is not allowed.

Salt may be stored as part of the encoded verifier output or as verifier metadata, depending on the selected library. It is not a secret, but it must be generated by the hashing implementation or a cryptographically secure random generator. Future implementation should target a salt length that meets current NIST guidance and the selected algorithm's recommended practice.

## Pepper Policy

Pepper is optional defense in depth. It is not a substitute for a strong password hashing algorithm, unique salt, rate limiting, or secure credential handling.

If Settleora uses a pepper later:

- The pepper secret must be stored outside PostgreSQL, such as in a deployment secret manager, hardware-protected store, or equivalent platform secret facility.
- The pepper secret must not be stored in `local_password_credentials`, migrations, seed data, logs, OpenAPI examples, appsettings committed to source control, or generated clients.
- Non-secret pepper metadata, such as a pepper key identifier or policy version, may be stored in `password_hash_parameters` if required for verification and rotation.
- Pepper rotation must be explicitly designed. Since the plaintext password is required to recreate a verifier, rotation normally happens on successful sign-in while both old and new pepper material are available, or by forcing affected users through password reset after old pepper material is retired.
- If a pepper is suspected to be compromised, affected credentials should be marked for rehash or reset according to the future incident policy, and the event must be auditable.

## Work Factor Policy

Work factors must be configured by password hashing policy, not hardcoded inside endpoint handlers.

Future auth workflow work must:

- Tune Argon2id memory, iteration, and parallelism parameters on deployment-class hardware.
- Tune PBKDF2 iteration count on deployment-class hardware if FIPS-compatible fallback is required.
- Keep verification latency high enough to slow offline attacks but low enough to avoid degrading normal sign-in or amplifying denial-of-service risk.
- Store the effective work factor and policy version with each credential.
- Support future work-factor increases without invalidating existing credentials immediately.
- Use `requires_rehash` to mark credentials that should be upgraded after a policy change.

The current schema already supports multiple parameter profiles during migration because each row stores algorithm, version, parameters, status, and rehash state.

## Verification Flow Boundaries

Verification must stay inside an API/domain auth service boundary. Endpoint handlers should not parse verifier strings, select algorithms, compare hashes, or log credential material directly.

Verification policy:

- Load the active local credential for the account through the approved auth boundary.
- Refuse disabled or revoked credentials before creating sessions.
- Verify the supplied password using the selected hashing implementation and the stored verifier metadata.
- Use constant-time comparison when the implementation exposes raw derived bytes or when Settleora must compare verifier material directly.
- Prefer the selected library's verified password-check API when it safely handles salt, work factor, and constant-time comparison internally.
- Do not log plaintext passwords, password hashes, salts, pepper identifiers in sensitive contexts, verifier strings, failed password values, or derived key material.
- Rate limiting, lockout, credential-stuffing defense, account enumeration behavior, and sign-in audit detail belong in future auth runtime design, not in the internal hashing boundary.

## Rehash And Rotation

Rehash is a normal migration path, not an exceptional failure state.

Future runtime should set or honor `requires_rehash` when:

- The policy version changes.
- The work factor increases.
- The selected algorithm changes.
- Pepper metadata changes.
- A library migration requires a new verifier encoding.

On successful sign-in with an active credential, future runtime may rehash the submitted password using the current policy, update the stored verifier output and metadata, clear `requires_rehash`, and emit a safe audit event. Failed verification must not rehash anything.

Disabled credentials must not authenticate but may be preserved for administrative review according to retention policy. Revoked credentials must not authenticate and should record `revoked_at_utc`. Account disablement, account deletion, credential revocation, password reset, and session revocation remain distinct state transitions and must be handled explicitly by future auth runtime design.

## Operational And Security Boundaries

- Plaintext local-account passwords must never be stored.
- Reversible encryption must not be used for local-account password storage.
- Raw reset tokens, recovery codes, passkeys, MFA secrets, raw session tokens, and OIDC tokens do not belong in `local_password_credentials`.
- Password hashing inputs, outputs, verifier strings, salts, peppers, and derived material must not be exposed through API responses, generated clients, logs, metrics, traces, audit metadata, or validation output.
- Credential creation, password change, password reset, credential disablement, revocation, rehash, and suspected compromise events require safe auth audit events when auth workflows exist.
- Workers and clients must not mutate local password credential rows directly. The API owns credential writes.
- Production startup must not silently migrate or rewrite password verifiers.

## Non-goals

This document does not authorize:

- Migrations.
- Credential creation implementation.
- Login implementation.
- Session middleware or token issuance.
- OpenAPI changes.
- Generated client changes.
- UI behavior.
- Runtime behavior changes.
